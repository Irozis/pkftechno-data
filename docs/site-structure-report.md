# pkftechno-data: site structure report

This report is an audit snapshot of the current `pkftechno-data` repository. It does not propose rewriting project logic, rebuilding the Tilda page from scratch, renaming slugs, or using GitHub Pages product previews as production product pages.

## 1. Repository structure

Important files and folders:

```text
data/
  products.json
scripts/
  build-product-pages.mjs
  convert-xlsx-to-json.mjs
  sync-pss2p-images-from-tilda.mjs

```

### Tilda production site

The production public site remains on Tilda: `https://pkftechno.ru`.

The current documented product URL pattern in this repository is:

```text
/product?slug=<slug>
```

Tilda should contain an existing product page / template, a hero Zero Block with assigned CSS classes, and T123 blocks from `tilda/`. The page should load data from GitHub Pages JSON and render the selected product by `slug`.

### GitHub Pages data source

GitHub Pages is the data and asset source. The current JSON URL used by `tilda/product-loader.html` is:

```text
https://irozis.github.io/pkftechno-data/data/products.json
```

The main data file is `data/products.json`. Tilda snippets, images, and helper scripts are stored in the repository and served statically after GitHub Pages deployment.

### GitHub Pages technical preview

The `products/` folder contains generated static HTML pages:

```text
products/<slug>/index.html
products/index.html
```

These pages are a technical preview only. They are generated by `npm run build:pages` from `data/products.json`. They should not be used as the production links from Tilda catalog cards.

## 2. Package scripts

Current `package.json` scripts:

| script name | command | what it does | production or helper |
|---|---|---|---|
| `validate` | `node scripts/validate-data.mjs` | Validates `data/products.json`: JSON parse, required arrays, unique product slugs, references from characteristics/images/related products. | Helper, should be run before publishing data changes. |
| `convert` | `node scripts/convert-xlsx-to-json.mjs ./source/products.xlsx` | Converts Excel sheets into `data/products.json`. | Helper/import tool. Use only when updating data from Excel. |
| `sync:pss2p-images` | `node scripts/sync-pss2p-images-from-tilda.mjs` | Reads product-card images from `https://pkftechno.ru/pss2ptest` and updates PSS-2P `product.image_url` plus `images[]` rows. | Helper/data maintenance. |
| `build:pages` | `node scripts/build-product-pages.mjs` | Generates static preview pages under `products/`. | Technical preview only, not production Tilda site. |

Recommended before publishing data changes:

```bash
npm run validate
```

When PSS-2P card images need to be refreshed from Tilda:

```bash
npm run sync:pss2p-images
npm run validate
```

`build:pages` is optional and only regenerates GitHub Pages preview pages. It is not required for the real Tilda product page to work.

## 9. Product Loader audit

File: `tilda/product-loader.html`.

What it does:

- Reads slug with `new URLSearchParams(window.location.search).get('slug')`.
- If slug is missing, shows `Товар не найден`.
- Fetches `https://irozis.github.io/pkftechno-data/data/products.json` with `{ cache: 'no-store' }`.
- Finds product by `item.slug === slug`.
- If product is not found, shows `Товар не найден`.
- Resolves images by checking `data.images` first, sorted by `sort`, then falling back to `product.image_url`, then an inline SVG placeholder.
- Inserts text into elements with classes such as `js-product-title`, `js-product-short`, `js-product-meta-series`, etc.
- Inserts image either into an `<img>` element or as background image on `.tn-atom`.
- Does not fail if an optional target element is absent; `querySelectorAll` simply returns an empty list.
- Creates `window.pkfProductContext = { product, data, images, characteristics, relatedProducts }`.
- Dispatches `document.dispatchEvent(new CustomEvent('pkfProductLoaded', { detail: window.pkfProductContext }))`.

Potential risks:

- CORS: GitHub Pages must allow Tilda to fetch `products.json`. Static GitHub Pages usually works, but this must be tested on the published Tilda page.
- Cache: GitHub Pages and Tilda/browser cache may delay visible updates. Loader uses `cache: 'no-store'`, but deployment caches can still exist.
- Missing CSS classes: if a Tilda Zero Block element does not have the expected class, that part of the hero will not update.
- Tilda preview can behave differently from the published page, especially for scripts and cross-origin fetches.

## 10. Product Specs audit

File: `tilda/product-specs-block.html`.

What it does:

- Listens for `pkfProductLoaded`.
- Also checks `window.pkfProductContext` in case the event already fired.
- Reads characteristics from `context.data.characteristics`.
- Filters rows by `item.product_slug === product.slug`.
- Sorts rows by `Number(item.sort)`.
- Groups display by `item.group`; empty group becomes `Характеристики`.
- Shows all rows if count is `<= 10`.
- If count is `> 10`, initially shows first 10 rows and a button `Показать все характеристики ↓`.
- Button toggles expanded state and changes text to `Скрыть характеристики ↑`.
- If there are no characteristics, shows `Характеристики уточняются по запросу.`
- Mobile adaptation: desktop uses two columns, around 959px becomes vertical, at 479px characteristic rows become vertical name/value blocks.

CSS prefix/classes:

```text
pkf-passport-*
productPassportSpecs
productPassportToggle
```

## 12. Product Related audit

File: `tilda/product-related-block.html`.

What it does:

- Listens for `pkfProductLoaded`.
- Also checks `window.pkfProductContext`.
- Reads `data.related_products`.
- Filters links by current `product.slug`.
- Finds related product records by `related_slug`.
- Uses product images from `data.images`, fallback `product.image_url`, fallback placeholder.
- Hides the whole block if no related products exist.

Current related link pattern in code:

```js
'/product?slug=' + encodeURIComponent(item.slug)
```

So the related cards currently link to Tilda `/product?slug=<slug>`, not GitHub Pages preview URLs. If the intended production route becomes `/pss2ptest?slug=<slug>`, this file would need a future intentional change, but this audit does not change it.

## 16. Potential mismatches and error sources

- `/product?slug=...` vs `/pss2ptest?slug=...`: current repository snippets and docs use `/product?slug=...`; some task text mentions `/pss2ptest?slug=...`. This should be resolved before changing production links.
- GitHub Pages preview vs Tilda production: `/products/<slug>/` exists only for preview; public cards should not link there.
- PP slugs: current slugs are `shk-pp1-pss-2p`, `shk-pp2-pss-2p`, `shk-pp3-pss-2p`; do not insert an extra dash.
- Tilda catalog cards may still contain old links from the source page. They should be updated to the current production Tilda pattern.
- Product Loader expects specific CSS classes. Missing classes will cause only that field to remain unchanged.
- Images exist in JSON, but Tilda must have `js-product-image` or `js-product-gallery` targets to display them.
- Characteristics exist in JSON, but `product-specs-block.html` should be below Product Loader or able to read `window.pkfProductContext`.
- GitHub Pages/Tilda/browser cache can delay updates.
- Tilda preview mode may not exactly match published behavior for cross-origin `fetch()`.
